> 本文档整理产品定位、内容架构、技术选型与实施节奏，**暂不实施**，供讨论与决策用。

## 一、网站定位与核心价值

| 维度 | 说明 |
|------|------|
| **定位** | 专注于 GLP-1 药物（司美格鲁肽等）的 **「决策引擎 + 避坑指南」** |
| **核心痛点** | 解决美国用户「药价贵、保险拒、常缺货、信息乱」四大难题 |
| **差异化壁垒** | 依靠 **实时动态数据（价格/库存）** 与 **真实用户评价（UGC）**，在 AI 泛滥时代提供可验证、可信任的价值 |

## 三、技术架构（全栈前端 / Serverless）

### 3.1 技术栈

| 层级 | 选型 | 用途 |
|------|------|------|
| 前端 | Next.js + Tailwind CSS | 页面、SEO、Core Web Vitals |
| 托管/运行环境 | Vercel | 部署、Cron、Edge |
| 数据存储 | Supabase | 药房排名、价格、用户评论、关系型数据 |
| 用户与权限 | Clerk | 注册、登录、订阅画像 |
| 自动化 | Vercel Cron Jobs | 定时更新药房库存/价格、FDA 状态等 |
| SEO | 结构化 Schema、Core Web Vitals | 五星好评、价格标签等富结果 |

### 3.2 架构说明

- **无自建服务器**：后端能力由 Supabase、Clerk、Vercel 等 Serverless/BaaS 提供。
- **「纯前端」的边界**：若「不要后端」指不跑自托管 Node/Python 服务，当前方案符合；若指连 BaaS 都不用，则需收缩范围（如暂不做登录、不做 Cron）。

## 五、方案补充与实施节奏建议

### 5.1 需要先定的三条线

1. **第一版是否要「后端能力」**
   - 要：保留 Supabase + Clerk + Vercel Cron，按「前端为主 + Serverless 后端」推进。
   - 不要：暂不做登录、不做 Cron，仅静态/客户端拉公开 API（如 FDA），工具以「输入→公式/区间结果」为主。

2. **数据来源与合规**
   - **价格/库存**：自爬（需考虑目标站 ToS、反爬）、合作方 API、或第三方数据商，先选定一种再设计 Cron 范围。
   - **FDA 缺货/批准**：优先用 FDA 官方 API/数据文件，合规且可持续。
   - **UGC 评论**：需审核策略、免责声明，建议在合规与反刷机制就绪后再开放。

3. **工具与内容的发布顺序**
   - 建议：**Calculator（不依赖实时数据）→ Quiz（逻辑+内容）→ Stock Alert（依赖数据源+用户系统）**。
   - 内容：先上平替百科、省钱攻略、合规教育等静态/半静态页，再上「实时看板」和 UGC。

### 5.2 分阶段实施节奏（建议）

| 阶段 | 内容与工具 | 数据与系统 |
|------|------------|------------|
| **Phase 1** | 平替百科、省钱攻略、合规教育、申诉信模板；**Out-of-pocket Calculator**（基于用户输入，不给实时报价） | 静态/半静态内容；可选 Supabase 存结构化内容；可不做登录 |
| **Phase 2** | **Alternative Finder Quiz**；补货提醒入口；「缺货/价格」看板 | 接入 1–2 个合规价格/库存数据源；Clerk 登录；Cron 更新 FDA + 价格/库存 |
| **Phase 3** | UGC 评论、历史价格曲线、更细的 Affiliate/Lead Gen 落地 | 审核与合规就绪；Supabase 评论与历史数据模型 |

### 5.3 技术落地注意点

- **Supabase**：尽早定好「药品 / 药房 / 价格快照 / 用户」等表结构，便于后续加评论、标签、历史曲线。
- **Clerk**：若后续要与 Supabase 中的用户画像/标签深度打通，需确认 webhook 与元数据是否满足。
- **Cron**：优先做「有限范围、可解释」的更新（如 FDA 状态、合作方 API），再考虑任何爬取策略，并确保合规。

### 5.4 合规与变现顺序

- 第一版可侧重「内容 + 免费工具」建立权威与流量；Affiliate/Lead Gen/Newsletter 在文案与合规确认后再明显放大，有利于后续卖站时的「合规、可持续」叙事。
